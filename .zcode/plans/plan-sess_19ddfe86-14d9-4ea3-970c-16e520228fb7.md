# Implementation Plan: Correct Upcast Rolling

## Overview

Replace the manual dice-rolling workflow for spells with an automated system where clicking a Cast button at a specific level resolves the correct scaled dice, rolls them, displays results, and consumes the slot — all in one action. Upcasting is driven by structured data, not by parsing spell description prose.

---

## Files to Create

### 1. `src/lib/spellEffects.ts` — Shared Spell Effect Resolver (NEW)

Core module containing:

**Types:**
- `SpellScaling`: discriminated union (`per-slot-level` | `slot-level-table` | `none`)
- `SpellDamageEffect` / `SpellHealingEffect` / `SpellEffect`
- `ResolvedSpellEffect`: the output of resolution (dice string + metadata)
- `DiceRollResult`: `{ expression, rolls, modifier, total }`
- `ResolvedSpellRoll`: discriminated union (damage with save info | healing)

**Functions:**
- `parseSimpleDice(expression: string): ParsedDice | null` — parses `NdX±M` format
- `scaleDicePerSlotLevel(baseDice, addedDice, additionalLevels): string` — combines dice counts when sides match
- `resolveSpellEffects(spell: SpellData, castLevel: number): ResolvedSpellEffect[]` — the central resolver that applies scaling rules to produce final dice expressions
- `rollResolvedEffect(effect: ResolvedSpellEffect): ResolvedSpellRoll` — rolls dice for a single resolved effect
- `resolveAndRollSpell(spell, castLevel): ResolvedSpellRoll[]` — convenience: resolve + roll all effects
- `formatSpellCastResult(spell, castLevel, rolls): string` — formats results for display

**Implementation details:**
- Effects are looked up from a scaling registry (imported from `spellScaling.ts`)
- For `per-slot-level`: `additionalLevels = max(0, castLevel - startsAboveLevel)`, then `baseDice + (additionalLevels × dicePerLevel)` combined into a single expression
- For `slot-level-table`: look up by cast level, fall back to base
- For `none`: return base dice unchanged
- Cantrips (level 0) never apply slot-based scaling
- When dice sides don't match between base and added dice, leave as separate expressions

## Key Design Decisions

1. **Separate scaling registry** rather than modifying spells.json deeply: keeps the large static file intact, allows incremental migration, and the registry can be code-generated later.

2. **Lightweight `higherLevel` bridge field** on SpellData: provides a simple fallback for simple per-level scaling without requiring every spell to be in the full registry.

3. **Roll-then-consume pattern**: Effects are resolved and rolled BEFORE the slot is consumed. If resolution fails (bad data), the slot is not spent.

4. **Reuse existing dice infrastructure**: All rolls go through `props.onRoll()` / `props.onRollPool()` which triggers the 3D dice animation overlay and console logging — no new dice system needed.

5. **No description text parsing for upcasting**: The resolver ONLY reads structured data from the scaling registry or the `higherLevel` bridge field. The description text is displayed but never parsed for scaling logic.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Not all 168 scaling spells get structured data in this pass | Categorize all, add data for simple cases (~60 spells), mark complex ones for manual review |
| Dice animation is async but slot consumption is sync | Generate dice values synchronously, trigger animation in parallel, consume slot immediately |
| Spell data format from JSON doesn't support TS union types natively | Use type assertion when reading; the bridge field is simple JSON-compatible |
| Attack roll spells need different workflow | Phase G handles these; Burning Hands (save-based) works immediately |
