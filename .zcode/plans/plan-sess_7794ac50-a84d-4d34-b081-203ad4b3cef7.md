## DM-First Onboarding & Campaign-Scoped Roles — Implementation Plan

### Phase 1: Campaign summaries with role data (§7-8)

The foundation — every campaign card needs to know the user's role and character.

**1a. Extend `CampaignSummary`** in `campaignStore.ts`:
- Add `myRole: "dm" | "player"` — derived from `dmUserId === userId`
- Add `myCharacterId: string | null` and `myCharacterName: string | null` — from the membership row's `character_id` join
- Update the SQL in `listCampaigns` to LEFT JOIN `characters` for the name

**1b. Group campaigns in CampaignPanel.tsx**:
- Split the flat list into "CAMPAIGNS I RUN" and "CAMPAIGNS I PLAY IN" sections
- Each card shows: name, role badge (DM / character name), member count, join code
- Empty state for each section when no campaigns exist

### Phase 2: Neutral first-run onboarding (§3-4)

**2a. Replace forced CharacterStartPanel with a choice screen:**
- When `characters.length === 0` AND no campaign is open, show a neutral choice screen instead of `CharacterStartPanel`
- Four options: Create a Character, Run a Campaign, Join a Campaign, Go to My Ledger
- "Create a Character" opens the existing Standard/Quickbuilder/Premade flow
- "Run a Campaign" opens a simple campaign creation form (name input → create → route to Table)
- "Join a Campaign" opens the CampaignPanel in join mode
- "Go to My Ledger" dismisses the screen, showing the empty vault + campaign list

**2b. "Run a Campaign" workflow:**
- Simple form: campaign name input + "Create Campaign" button
- Creates campaign and opens the Table immediately (using existing `createCampaign` + `setActiveCampaign` + `setCampaignOpen`)
- No character creation required
- The DM has `characterId: null` — party strip shows "DM" with no character

### Phase 3: Join improvements + eligibility (§10-14)

**3a. "Create a Character" button in join flow:**
- When the user has no characters, show a "Create a Character" button instead of just the muted text
- The button opens the standard builder; after creation, return to join flow with the code preserved and new character selected

**3b. Character eligibility checking:**
- Before the join dropdown, split characters into "Available" and "Already in a campaign"
- Characters enrolled in another campaign are shown with the campaign name but cannot be selected
- Server-side: `joinCampaign` checks `SELECT 1 FROM campaign_members WHERE character_id = ? AND campaign_id != ?` and rejects if found
- "Duplicate for this campaign" button: creates a copy of the character via `POST /api/characters` with the same data but a new name suffix (e.g., "Rook, Hollow Crown")

### Phase 4: Verification
- `npm run lint` + `npm run build` + `npm test` clean
- Two-profile test: register new account, choose "Run a Campaign", create without character, confirm Table opens, invite player
- DM of Campaign A joins Campaign B as player: confirm both show correctly grouped

### Out of scope (this round)
- Co-DM / spectator roles
- Switching campaign character mid-campaign (already partially exists via the dropdown in CampaignPanel detail view)
- Create-character-and-return-to-join flow (deferred — the "Create a Character" button with code preservation is done, but the full roundtrip requires builder integration that's risky for this round)
